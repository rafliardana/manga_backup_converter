# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Manga Backup Converter ‚Äî a Flutter/Dart monorepo that converts manga backup files between formats: Aidoku (.aib), Paperback (.pas4), Tachi (.tachibk, .proto.gz), Tachimanga (.tmb), and Mangayomi (.backup). Currently a standalone CLI; Flutter app is planned but not yet functional..

## Monorepo Structure (Melos)

- Root: Flutter app + monorepo workspace config (melos scripts, dependency_overrides)
- `packages/mangabackupconverter_cli/` ‚Äî Core conversion logic (pure Dart, no Flutter dependency)
- `packages/aidoku_plugin_loader/` ‚Äî Aidoku WASM plugin loader (host imports, isolate management, picks runtime)
- `packages/wasm_runner/` ‚Äî Abstract WASM runner interface (`WasmRunner`, `WasmRuntimeException`, `WasmTrapException`)
- `packages/wasmer_runner/` ‚Äî Wasmer FFI runner implementing `WasmRunner` (native platforms, build hook downloads prebuilt wasmer)
- `packages/web_wasm_runner/` ‚Äî Browser WebAssembly runner implementing `WasmRunner` (web platform)
- `packages/wasm3/` ‚Äî Wasm3 interpreter runner implementing `WasmRunner`
- `packages/quickjs` ‚Äî QuickJS JS engine (git dep: `getBoolean/quickjs-dart`, branch `fix/pinnedDart`; downloads prebuilt binaries via build hook)
- `packages/app_lints/` ‚Äî Shared lint rules (based on `solid_lints`)
- `packages/jsoup/` ‚Äî Jsoup-compatible HTML parsing (Rust scraper on Windows/Linux/Android, SwiftSoup on iOS/macOS, TeaVM on Web)
- `packages/scraper/` ‚Äî Rust scraper crate (html5ever/Servo) FFI bridge; build hook compiles via cargo + registers CodeAsset; for Android, auto-discovers NDK from `ANDROID_HOME/ndk/` and runs `rustup target add` before cross-compiling
- `packages/swiftsoup/` ‚Äî SwiftSoup FFI bindings for iOS/macOS; @objc bridge wrapper compiled from Swift via SPM; build hook compiles Swift and registers CodeAsset; Dart bindings generated by ffigen from auto-extracted ObjC header; `SwiftSoupParser` lives in `packages/jsoup/lib/src/swift/` (mirrors scraper pattern)
- `packages/assets/` ‚Äî Asset code generation
- `packages/constants/` ‚Äî App-wide constants

## Architecture

**Riverpod App Architecture** ‚Äî feature-first with layered structure per feature:

```txt
lib/src/features/<feature>/
  ‚îú‚îÄ‚îÄ domain/         # Entities, value objects (freezed + modddels)
  ‚îú‚îÄ‚îÄ data/           # Repositories, DTOs (dart_mappable), data sources
  ‚îú‚îÄ‚îÄ application/    # Riverpod providers/controllers (riverpod_generator)
  ‚îî‚îÄ‚îÄ presentation/   # ConsumerWidgets, screens
```

Active features: `books`, `connectivity`, `initialization`, `settings`. The `example_feature` directory is a reference template showing the full pattern.

### CLI Package Architecture

`packages/mangabackupconverter_cli/` is organized by format:

- `lib/src/converter.dart` ‚Äî `MangaBackupConverter` class with import methods per format
- `lib/src/common/` ‚Äî `Convertable` interface, shared mappers, extension repo index
- `lib/src/formats/<format>/` ‚Äî Format-specific backup models and parsers
- `lib/src/pipeline/` ‚Äî Migration pipeline API (BackupFormat, MangaSearchDetails, MigrationPipeline, plugin sources)
- `lib/src/commands/` ‚Äî CLI command implementations + interactive terminal UI (`terminal_ui.dart`, `migration_dashboard.dart`, `live_search_select.dart`, `manga_details_screen.dart`, `extension_select_screen.dart`)
- CLI TUI screens use `TerminalContext` + `ScreenRegion` for all I/O, NOT `print()` ‚Äî `print()` goes through Dart zones while `TerminalContext.write` does not, enabling zone-based log redirection in interactive mode
- `TerminalContext` bundles KeyInput, ScreenRegion output, SIGINT handling, and terminal dimensions ‚Äî screens receive it as a parameter, never create their own I/O objects
- `ScreenRegion.render` calls `writeln` for every line ‚Äî NEVER pad output to `context.height` lines; the final `writeln` scrolls the terminal past the bottom, corrupting the TUI; let `clearDown()` handle leftover space (no other TUI screen pads)
- TUI scroll indicators ("‚Üë more above" / "‚Üì more below") consume lines from the viewport budget ‚Äî subtract them from the line budget BEFORE packing entries, or the output overflows the terminal
- Variable-height TUI entries (e.g. wrapped detail lines): compute `maxVisible` by packing entries into a line budget from `scrollOffset`, not by dividing available lines by a fixed entry height
- `TerminalContext` is single-use per process ‚Äî `dispose()` cancels `_stdinSub` (broadcast source), permanently killing stdin and preventing event loop hangs; use one context for the entire interactive session
- `readLineSync()` must be called BEFORE creating `TerminalContext` ‚Äî once the context puts stdin in raw mode (`lineMode=false`), `readLineSync()` won't work; use `_readYesNo` for y/N prompts and `_readLine` (in `convert_command.dart`) for free-text input after context creation ‚Äî both read raw keys from `KeyInput`
- Raw mode output: when `TerminalContext` is active, use `\r\n` (not `\n`) for newlines in `context.write()` calls ‚Äî raw mode disables automatic CR insertion
- `TerminalContext.test()` constructor accepts `StringSink` + `Stream<List<int>>` for testable rendering/input
- `MangaDetailsScreen.run()` returns `Future<bool>` ‚Äî `true` = Enter (confirm selection), `false` = Escape (go back); `LiveSearchSelect` closes the event loop on `true` (same as direct Enter on a result)
- `ExtensionSelectScreen` Enter behavior: `when cursorIndex >= 0` guard ‚Äî Enter does nothing while search bar is focused; with extensions toggled returns toggled set, without toggled auto-selects highlighted entry
- CLI TUI hyperlinks: wrap text in `green()` then `hyperlink()` ‚Äî `green(text)` not `green(bold(text))` to match the dashboard's color; bold brightens green to a different shade
- `renderMarkdown()` in `terminal_ui.dart` converts markdown (`[text](url)`, `**bold**`, `*italic*`) to ANSI escapes ‚Äî applied per-line AFTER `wordWrap()` to avoid ANSI sequences spanning wrapped lines
- `win_console_stub.dart` / `win_console_native.dart` ‚Äî conditional import (`dart.library.ffi`) enables `ENABLE_VIRTUAL_TERMINAL_INPUT` on Windows; `enableVirtualTerminalInput()` called in `KeyInput.start()`, `restoreConsoleMode()` in `dispose()`
- `SearchInputState` in `terminal_ui.dart` owns query text, cursor position, focus state (`focused`), key routing (`tryHandleKey` ‚Üí `SearchKeyResult`), and box rendering (`renderBox`) ‚Äî callers pass unhandled keys through and check the result enum; `renderSearchInput()` renders the ANSI inverse block cursor
- Both search screens use `cursorIndex` (`-1` = search bar, `>= 0` = result) and sync `searchInput.focused` on arrow navigation; `tryHandleKey` auto-refocuses on text-modifying keys (CharKey, Backspace, Delete) but leaves Space to callers for screen-specific actions
- `KeyInput` parses two escape sequence formats: `ESC [ <byte>` (arrows, Home, End) and `ESC [ <digit> ~` (Delete=3); the escape buffer handles split delivery across both
- `KeyInput` enables SGR extended mouse mode (`\x1b[?1000h\x1b[?1006h`) on `start()` and disables on `dispose()` ‚Äî scroll wheel events parsed as `ScrollUp` (button 64) / `ScrollDown` (button 65); all other mouse events silently discarded
- TUI screens with a search bar (`cursorIndex = -1`) must reset `scrollOffset` to `0` when cursor enters the search bar ‚Äî otherwise viewport stays scrolled, showing stale "‚Üë more above" indicator
- `KeyInput._controller` is broadcast ‚Äî parent screens `cancel()` their subscription before opening a child screen, child creates its own subscription, then parent creates a fresh subscription on return (declare `keySub` as non-final). Do NOT use `pause()`/`resume()` (causes buffered event flood) or `suspend()`/`start()` (removed)
- `convert_command.dart` interactive callbacks (e.g. `onSelectExtensions`, `onConfirmMatches`) follow a stop/restore pattern: `spinner.stop()` ‚Üí `loadingRegion.clear()` ‚Üí `context.showCursor()` ‚Üí run TUI screen ‚Üí `context.hideCursor()` ‚Üí `spinner.start(...)`. Non-interactive mode uses auto-select fallbacks.
- `convert_command.dart` uses `runZoned` with `ZoneSpecification.print` to redirect all `print()` (including from `aidoku_plugin_loader`) to a log file when interactive; `Zone.root.print()` escapes the zone for user-facing messages
- `_readPath` in `convert_command.dart` hides the terminal cursor on entry (`_renderPathInput` draws its own inverse-video block cursor) and restores it on all exit paths ‚Äî callers should not call `showCursor()` before `_readPath`
- `PathInputState.handleKey()` exhaustively matches all `KeyEvent` subtypes ‚Äî no `default` case needed; adding new `KeyEvent` subtypes requires adding a case here
- `extension_entry.dart` ‚Äî `ExtensionEntry` sealed class (`id`, `name`, `languages`, `contentRating` (0=Safe, 1=Suggestive, 2=NSFW), `cacheKey`); `AidokuExtensionEntry` adds `version`, `altNames`, `iconUrl`, `downloadUrl`, `baseUrl?` (`cacheKey` = `'$id-v$version'`); `StubExtensionEntry` has only base fields (`cacheKey` = `id`)
- `PluginLoader` base class has `downloadPluginBytes(entry)` and `loadPluginFromBytes(entry, bytes)` with default null-returning implementations ‚Äî `AidokuPluginLoader` overrides both; `CachingPluginLoader` uses these to cache/restore plugin bytes format-agnostically
- `lib/src/pipeline/plugin_source_stub.dart` ‚Äî `StubPluginSource` implements `PluginSource`; must be updated when the interface changes
- `lib/src/pipeline/source_manga_data.dart` ‚Äî `SourceMangaData` normalized type (chapters, history, tracking, categories)
- `lib/src/pipeline/target_backup_builder.dart` ‚Äî `TargetBackupBuilder` sealed class; `AidokuBackupBuilder` is the only concrete impl; `build()` accepts optional `sourceFormatAlias` for backup metadata
- **Postcard integer encoding**: `u8`‚Äì`u64` ‚Üí unsigned varint (LEB128); `i8`‚Äì`i64` ‚Üí zigzag varint; `f32` ‚Üí 4 LE bytes; `f64` ‚Üí 8 LE bytes. Both `PostcardReader.readI64` and `PostcardWriter.writeI64` use zigzag varint, NOT raw bytes
- **Postcard zigzag web safety**: JS bitwise ops work on 32-bit signed ints ‚Äî use arithmetic (`isOdd`, `~/ 2`, `* 2`) not bitwise; see `postcard_reader.dart`/`postcard_writer.dart`
- **Wasm3 code assets**: `packages/wasm3/hook/build.dart` compiles vendored wasm3 C source via `native_toolchain_c` and registers it as a `CodeAsset` with `DynamicLoadingBundled()` ‚Äî runs automatically during `dart run`, `dart build`, `dart test`; no manual install needed; defines `M3_HAS_TAIL_CALL=0` for Android armv7 ‚Äî NDK clang 18 crashes on `musttail` attribute
- **Wasm3 version**: vendored at latest HEAD (`79d412e`, post-v0.5.0); `m3_emit.c`, `m3_emit.h`, `m3_optimize.c` were removed upstream (merged into `m3_compile.c`) ‚Äî `hook/build.dart` reflects this; regenerate bindings with `dart run ffigen --config ffigen.yaml` from `packages/wasm3/`
- ffigen config in `packages/wasm3/ffigen.yaml`; uses `ffi-native` mode ‚Äî generates `@Native`-annotated top-level functions
- `Wasm3Runner` `readMemory`/`writeMemory`/`call` throw `WasmRuntimeException` (an `Exception`, not `Error`)
- `WebWasmRunner.call` throws `WasmRuntimeException` for missing exports ‚Äî NOT `ArgumentError`
- `MangaConverterException`, `WasmRuntimeException`, `WasmTrapException`, `AixParseException`, `SourceListFetchException` all have optional `cause` and `stackTrace` fields ‚Äî use `Error.throwWithStackTrace` when re-wrapping exceptions to preserve the original stack trace
- **WASM runner conditional export**: `aidoku_plugin_loader/lib/src/wasm/wasm_runner.dart` re-exports `wasm3` on native, `web_wasm_runner` on web via `if (dart.library.js_interop)`
- Root `pubspec.yaml` must depend on packages with build hooks (wasm3, scraper, swiftsoup, quickjs) for `dart test` from root to discover code assets
- **No `print()` in WASM isolate code** ‚Äî `aidoku_host.dart`, `wasm_isolate.dart`, and the active WASM runner route all log messages through `onLog` callback (threaded via `buildAidokuHostImports` and `WasmRunner.fromBytes`), which sends `WasmLogMsg` to the main isolate; this allows `convert_command.dart`'s `runZoned` print redirect to capture them
- `_processCmd` in `wasm_isolate.dart` replies via `cmd.replyPort.send()` ‚Äî when changing reply format (e.g. to a tuple), update ALL corresponding `await port.first as ...` casts in `aidoku_plugin_io.dart`
- **Isolate error handling**: `wasmIsolateMain` and `_processCmd` use `on Object catch` (not `on Exception catch`) ‚Äî `Error` subtypes (`ArgumentError`, `RangeError`, `StateError`) from the WASM runner/FFI must be caught or the isolate dies silently, causing `port.first` hangs on the main isolate
- **WasmSemaphore platform backends**: Windows ‚Üí `CreateSemaphoreA` (kernel32); macOS/iOS ‚Üí GCD `dispatch_semaphore_*` (libSystem via `DynamicLibrary.process()`); Linux/Android ‚Üí POSIX `sem_init`/`sem_wait`/`sem_post` ‚Äî `sem_init` is deprecated and non-functional on macOS (returns -1, ENOSYS), so POSIX must NOT be used on Apple platforms
- **Isolate init failure**: Runner creation happens AFTER the handshake sends cmdPort ‚Äî if `WasmRunner.fromBytes` fails, the isolate enters a command-drain loop replying with errors (can't send on the already-consumed handshake port)
- **`_sendErrorReply` in `wasm_isolate.dart`**: Best-effort error reply for all command types; `WasmSearchCmd`/`WasmMangaDetailsCmd` reply with `(String error, List<String>)`, all others reply with `null`
- `aidoku_plugin_io.dart` (native) and `aidoku_plugin_web.dart` (web) are conditional exports of `aidoku_plugin.dart` ‚Äî any public method added to one MUST be added to the other
- Host import errors (CSS selector failures, HTML parse errors) are logged with `[CB]` prefix via `onLog` ‚Äî `wasm_isolate.dart` accumulates these during WASM calls and returns them as warnings alongside results; `PluginSearchPageResult.warnings` carries them to the pipeline; `_streamSearch` emits `PluginSearchError` for non-empty warnings; web worker `console.warn` calls in `wasm_worker_js.dart` must also use `[CB]` prefix for consistency
- WASM plugins are single-threaded ‚Äî never call `search`/`getMangaWithChapters` concurrently on the same `PluginSource`; the migration dashboard serializes searches (one manga at a time); `_streamSearch` emits raw search results only ‚Äî enrichment via `getMangaWithChapters` happens in `detailedConfirmations` (after user confirms matches) using `Future.wait` for concurrency
- `migration_dashboard.dart` search lifecycle: `activeEntry` tracks the entry being searched; `pendingRetries` queues re-selected entries; `startNextSearch` drains retries before the linear index scan ‚Äî when skipping a deselected entry in the scan, `searching` is set to `false`, so re-select logic must check `match == null` (not `searching`) to decide whether to retrigger
- `migration_dashboard.dart` auto-match: takes only first result per plugin, enriches all candidates via `onFetchDetails` on `_SearchDoneEvent`, then `_findBestMatch` picks candidate with most chapters (not title similarity); `_EnrichmentDoneEvent` carries enriched results back to the event loop
- `migration_dashboard.dart` Tab key: opens `MangaDetailsScreen` for the highlighted entry's match (view-only); does NOT cancel `activeSub` ‚Äî non-destructive unlike Enter which needs exclusive plugin access; reuses enriched `details`/`chapters` when available
- `diceCoefficient` in `terminal_ui.dart` ‚Äî bigram Dice coefficient (0.0‚Äì1.0) for title similarity; used by live search sorting
- `live_search_select.dart` display rows include non-selectable group headers (result index `-1`); scroll offset operates in display-row space, not result-index space
- `get_manga_update` WASM ABI: `(manga_descriptor_rid, needs_details, needs_chapters)` ‚Äî `getMangaDetails` accepts `{bool includeChapters = false}`; `plugin_source_aidoku.dart` passes `includeChapters: true` so chapters are included in migration output
- `PluginSearchResult` carries optional `pluginSourceName` (friendly display name, falls back to `pluginSourceId`), `details` (`PluginMangaDetails?`), and `chapters` (`List<PluginChapter>`) ‚Äî NOT populated during search; the TUI uses `onFetchDetails` callback when needed; `detailedConfirmations` in `_runMigration` fetches details for confirmed matches before building target backup; when constructing enriched copies, always preserve `pluginSourceName`
- WASM `get_search_manga_list` returns minimal data (authors typically empty); full author/artist data comes from `getMangaWithChapters` (called in `detailedConfirmations`, not during search) ‚Äî always prefer detail-level fields over search-level fields when both exist
- `PluginMangaDetails` has separate `authors` and `artists` lists ‚Äî display code should merge both into a `<String>{}` set to deduplicate (authors who are also artists)
- `lib/src/proto/` ‚Äî Protocol buffer schemas for Tachi forks (mihon, j2k, neko, sy, yokai)
- `lib/src/exceptions/` ‚Äî Format-specific exception classes

Each backup format class has a `fromData(Uint8List)` factory and conversion methods to other formats.

## Pipeline & Backup Format

- `BackupFormat` (sealed) is the single type for all format selection (CLI, pipeline, TachiBackup)
- Aliases: `aidoku, paperback, mihon, sy, j2k, yokai, neko, tachimanga, mangayomi`
- `BackupType` enum and `TachiFork` enum have been deleted ‚Äî do not recreate
- `TachiBackup.format` is `Tachiyomi` (sealed subtype of `BackupFormat`), not an enum
- `Mangayomi` extends `BackupFormat` directly, not `Tachiyomi`
- `BackupFormat` hierarchy uses dart_mappable `discriminatorKey`/`discriminatorValue` for serialization
- `TachiUpdateStrategy` uses `ValuesMode.indexed` ‚Äî maps to/from `int` (0=alwaysUpdate, 1=onlyFetchOnce), not string names
- `ConversionStrategy` (sealed): `DirectConversion` (Tachimanga‚ÜîTachi) or `Migration` (all other pairs including same-format); no `Skip` ‚Äî all pairs allow plugin migration
- `MigrationPipeline.run()` accepts `forceMigration` ‚Äî when true, `DirectConversion` pairs use plugin migration instead of direct conversion
- `TachiBackup.toTachimangaBackup()` flattens nested Tachi structures into normalized Tachimanga DB tables with autoincrement-style sequential PKs (matching Tachimanga's `IntIdTable`); history is collapsed to one row per manga (Tachimanga's History table has `UNIQUE(manga_id)`) ‚Äî latest chapter by `lastRead` timestamp, summed `readDuration`; Source IDs are preserved as-is (Source table uses `BIGINT NOT NULL`, not autoincrement)

### Pipeline Data Flow

- `MigrationPipeline.onConfirmMatches` is a batch callback ‚Äî receives all `SourceMangaData` plus `onSearch` (streaming) and `onFetchDetails` functions; UI handles searching and user interaction
- `PluginSearchEvent` (sealed) streams search results per-plugin: `PluginSearchStarted` (with optional `pluginName`) / `PluginSearchResults` (with optional `pluginName`) / `PluginSearchError`
- `MangaMatchConfirmation.sourceManga` is `SourceMangaData` (not `MangaSearchDetails`) ‚Äî carries chapters, history, tracking, categories from source backup
- `ConvertableBackup.sourceMangaDataEntries` extracts `List<SourceMangaData>` from each backup format
- `MangaMatchProposal.sourceManga` remains `MangaSearchDetails` (UI display only)
- `MangaSearchDetails` constructor applies `fixDoubleEncoding` (from `common/fix_double_encoding.dart`) to all text fields (`title`, `altTitles`, `authors`, `artists`, `description`) ‚Äî NOT a const constructor; do not use `const MangaSearchDetails(...)`
- `SourceChapter`, `SourceHistoryEntry`, `SourceTrackingEntry`, `PluginChapter` constructors apply `normalizeChapterNumber` (from `common/normalize_chapter_number.dart`) to chapter/volume numbers ‚Äî NOT const constructors
- `tachi_backup.dart` must directly import `tachi_backup_chapter.dart`, `tachi_backup_history.dart`, `tachi_backup_tracking.dart` ‚Äî they are NOT re-exported by `tachi_backup_manga.dart`

## Lint Rules

- Base: `solid_lints` via `packages/app_lints`
- `prefer_single_quotes`, `strict-inference: true`, `dart analyze --fatal-infos`
- `omit_obvious_local_variable_types` + `specify_nonobvious_local_variable_types` + `avoid_multiple_declarations_per_line` ‚Äî use `dart fix --apply <directory>` after writing new code (only accepts one path argument)
- `dart fix --apply` may add incorrect imports on web-only files (e.g. importing `dart:io` transitives) ‚Äî always review changes after running
- `avoid_catching_errors` lint: don't use `on UnimplementedError catch` in tests ‚Äî use `on Object catch (e)` + `check(e).isA<UnimplementedError>()` instead
- `directives_ordering` ‚Äî all `package:` imports must be in one alphabetically-sorted section; conditional imports (`if (dart.library.ffi)`) count as their primary URI for sorting
- **Directional UI required**: Use `EdgeInsetsDirectional`, `PositionedDirectional`, `AlignmentDirectional`, `BorderDirectional`, `BorderRadiusDirectional` instead of their non-directional counterparts
- CI enforces formatting via `melos run verify_format`

- **State**: Riverpod + riverpod_generator + hooks_riverpod

- **Web-only tests**: `@TestOn('browser')` + `dart test --reporter expanded --platform chrome test/web/` ‚Äî hand-crafted WASM binaries as `Uint8List` constants for testing without .aix fixtures

- **CLI formats**: protobuf, archive, sqflite_common

## CI

GitHub Actions runs on all branches: format verification, analysis, tests with coverage (Codecov), then platform builds (iOS, Android, Web, Windows, Linux). Flutter 3.41.1 stable.
- All `actions/checkout@v4` steps require `submodules: true` ‚Äî wasm3 vendored source is a git submodule at `packages/wasm3/vendor/wasm3`
- `test_with_coverage` defaults to VM service port 8181; concurrent `melos exec` causes port conflicts and hangs ‚Äî test scripts use `concurrency: 1`
- CI `test` job installs Rust toolchain (`dtolnay/rust-toolchain@stable`) on Windows/Linux before tests ‚Äî scraper build hook needs `cargo`; without it, the hook downloads rustup from scratch and hangs
- CI `test` job has `timeout-minutes: 30` as a safety net against hangs
- CI flutter build jobs run from repo root ‚Äî build artifacts are under `build/`

- `AidokuPlugin.fromAix()` accepts `defaults: Map<String,dynamic>` to pre-seed per-source defaults (e.g. auth tokens); keys are raw (no sourceId prefix); values are postcard-encoded to `Uint8List` (bool‚Üí`writeBool`, int‚Üí`writeSignedVarInt`, String‚Üí`writeString`); see WASM_ABI.md "defaults" section
- `HostStore.defaults` stores ONLY `Uint8List` values (postcard-encoded) ‚Äî no raw `int` or `String`; `defaults::get` adds as `BytesResource` and returns RID; `defaults::set` reads `[u32 len LE][u32 cap LE][data]` from WASM memory (same layout as `send_partial_result`)
- `setting_item.dart` `defaultEntry` getters return postcard-encoded `Uint8List` for all types (bool, int, string)
- `start` export is always generated by aidoku-rs `register_source!` macro (core, not optional); the `on Exception catch` around `runner.call('start')` is defensive only
- `HostStore` is shared between web (Worker) and native (WASM isolate) ‚Äî stores raw `Uint8List` resources
  - Decoding postcard results must happen at the consumer level (`aidoku_plugin_io.dart` / `aidoku_plugin_web.dart`), not inside `HostStore`
  - Native isolate forwards raw bytes via `WasmPartialResultMsg`
- `_encodeString` in `aidoku_host.dart` returns **raw UTF-8** (`utf8.encode`), NOT postcard-encoded ‚Äî aidoku-rs SDK reads host strings via `String::from_utf8(buffer)` which expects no length prefix
- **html:: WASM imports use jsoup OO API** ‚Äî re-integrated after switching from wasmer to wasm3 (no VEH conflict). `wasm_isolate.dart` creates `Jsoup()` per isolate with graceful fallback (`htmlParser == null` ‚Üí imports return `-1`). HTML imports in `imports/html_imports.dart`; `ImportContext` helpers in `libs/import_context.dart`. `HtmlElementResource`/`HtmlElementsResource` in `host_store.dart` store jsoup DOM objects.
- **aidoku_host.dart modular imports**: `buildAidokuHostImports` delegates to `imports/html_imports.dart`, `imports/std_imports.dart`, `imports/net_imports.dart`, `imports/env_imports.dart`, `imports/defaults_imports.dart`, `imports/canvas_imports.dart`, `imports/js_imports.dart`; shared context in `libs/import_context.dart`
- **quickjs-dart fork API**: `NativeJsEngine` (sync FFI) requires `NativeEngineManager()` first (initializes module-level `late _manager`); neither is exported from barrel ‚Äî use `// ignore: implementation_imports` on `package:quickjs/src/native_js_engine.dart`; `JsEngine`/`JsEngineManager` (async isolate-based) ARE exported but unsuitable for sync WASM host imports; `NativeEngineManager.length` tracks active engines, `.dispose()` frees the QuickJS runtime
- **`std::parse_date`** uses `package:intl` `DateFormat` when format string is provided (Unicode TR35/ICU patterns, same as Swift DateFormatter); falls back to ISO 8601 `DateTime.tryParse` when format is empty; handles `"UTC"`/`"GMT"` timezone
- **jsoup package layout** (`packages/jsoup/lib/src/`): `jsoup.dart` (entry point), `html_parser.dart` (abstract `NativeHtmlParser` interface), `jsoup_version.dart`; `nodes/` (OO API: `document.dart`, `element.dart`, `elements.dart`, `node.dart`, `text_node.dart`); `platform/` (`parser_native.dart`, `parser_web.dart` ‚Äî conditional import factories); `scraper/` (`scraper_parser.dart` ‚Äî Rust scraper backend, used on Windows/Linux/Android); `jni/` (`jni_parser.dart`, `jre_manager.dart`, `bindings/` ‚Äî JNI backend, retained for Android fallback/reference); `swift/` (`swift_parser.dart` ‚Äî SwiftSoup backend via `package:swiftsoup`, used on iOS/macOS); `web/` (`teavm_parser.dart`, `teavm_bundle.dart` ‚Äî TeaVM backend)
- **jsoup OO API** (`packages/jsoup/`): public classes `Jsoup`, `Document extends Element`, `Element extends Node`, `Elements extends Iterable<Element>`, `Node` (base class), `TextNode extends Node`; `NativeHtmlParser` is internal (not exported from barrel `package:jsoup/jsoup.dart`)
- `Jsoup.parser` is `@internal`; `Element.fromHandle`/`Elements.fromHandle`/`Document.fromHandle` are `@internal` ‚Äî external code uses `Jsoup()`, `jsoup.parse()`, `element.select()` etc.
- `Elements` is always native-backed; public constructor `Elements(Jsoup, List<Element>)` creates via `createElements`; for empty fallback use `Elements.fromHandle(parser, parser.createElements(const <int>[]))`
- `Node` base class: `nodeName`, `parentNode`, `childNode(index)`, `childNodes`, `childNodeSize`, `outerHtml`, `remove()`
- `TextNode extends Node`: `text` (get/set), `wholeText`, `isBlank`; `Element.textNodes` returns `List<TextNode>`, `Element.childNodes` returns `List<Node>` (mixed)
- `Jsoup` factory methods: `element(tag)`, `textNode(text)`, `elements(list)` ‚Äî preferred entry points alongside `parse`/`parseFragment`
- Node type discrimination in `jni/jni_parser.dart`: `_addNode()` checks `nodeName() == "#text"` then casts with `.as(TextNode.type)` or `.as(Element.type)`
- JNI inherited methods (not in jnigen bindings): use raw FFI `ProtectedJniExtensions.lookup` pattern ‚Äî see `_callBooleanMethodWithObject` for `List.add`, `_callIntMethod` for `List.size`
- **Rust scraper backend** (`packages/scraper/`): Rust `scraper` crate 0.25 (html5ever/Servo) compiled to cdylib via cargo; ~45 `extern "C"` functions with `scraper_` prefix; thread-local handle stores (`DOCUMENTS`, `NODES`, `NODE_LISTS`) with atomic counter; `ScraperParser` lives in `packages/jsoup/lib/src/scraper/` (not in scraper package) to avoid circular dependency
- **Scraper `:contains()` support**: `:contains()`, `:containsOwn()`, `:containsWholeText()`, `:containsWholeOwnText()`, `:containsData()`, `:matches(regex)`, `:matchesOwn(regex)` implemented via `contains_filter.rs` ‚Äî strips the pseudo-selector from the CSS string, passes base selector to Servo, then post-filters matched elements; `:matches`/`:matchesOwn` use `regex` crate
- **Scraper Jsoup parity**: `scraper_text`/`scraper_text_node_text` normalize whitespace (`split_whitespace().join(" ")`); `scraper_text_node_whole_text` returns raw text; `scraper_node_outer_html` for text nodes HTML-escapes `&`, `<`, `>`; `scraper_data` only returns content for `script`/`style`/`textarea`/`title` (empty for other elements); `scraper_class_name` returns `""` (not null) for classless elements; `scraper_node_abs_url` returns `""` for relative URLs when base URI is empty (absolute URLs still returned)
- **Scraper select self-matching**: `scraper_select`/`scraper_select_first` check if the element itself matches the selector before iterating descendants (via `sel.matches(&el_ref)`) ‚Äî Java Jsoup includes the root element in results; without this, `el.selectFirst("a")` on an `<a>` element returns null
- **Scraper mutation gotcha**: `scraper::node::Element` uses `OnceCell` for `id`/`classes` caching ‚Äî attribute mutations must rebuild the entire Element via `Element::new()` to invalidate caches (see `rebuild_element` in `mutation.rs`)
- **html5ever fragment wrapping**: `Html::parse_fragment()` wraps content in implicit `<html>/<head>/<body>` ‚Äî `collect_fragment_trees()` in `mutation.rs` unwraps these before transplanting children
- **Scraper build hook caching**: To force rebuild after Rust changes, delete BOTH `.dart_tool/hooks_runner/scraper/` (per-package hash/output cache) AND `.dart_tool/hooks_runner/shared/scraper/` (built DLL cache); deleting only the DLL causes `PathNotFoundException` because the hooks_runner skips re-running the hook if hashes match
- **SwiftSoup build hook caching**: Same pattern ‚Äî delete BOTH `.dart_tool/hooks_runner/swiftsoup/` AND `.dart_tool/hooks_runner/shared/swiftsoup/` to force rebuild after Swift source changes
- **quickjs build hook caching**: Hook cache key is the hook binary hash ‚Äî changing only `.github/workflows/asset_build.yaml` in quickjs-dart does NOT invalidate the cache; force re-download by copying a DLL from the release directly to `.dart_tool/lib/`, or make a trivial change in `hook/build.dart`
- **SwiftSoup cross-compilation**: macOS: `--triple arm64-apple-macosx10.15` / `x86_64-apple-macosx10.15`; iOS: `-Xswiftc -sdk -Xswiftc $(xcrun --sdk iphoneos --show-sdk-path)`; `swift build --show-bin-path` locates output dir
- **Windows build hook DLL race**: `PathExistsException` when `dart test` tries to copy native DLLs that already exist in `<package>/.dart_tool/lib/`; fix: `rm -rf <package>/.dart_tool/lib/` and retry
- **jsoup platform routing**: Windows/Linux ‚Üí Rust scraper (`ScraperParser`); Android ‚Üí Rust scraper; iOS/macOS ‚Üí SwiftSoup (`SwiftSoupParser`); Web ‚Üí TeaVM (`TeaVMParser`); `parser_native.dart` returns `SwiftSoupParser()` for iOS/macOS, `ScraperParser()` for all others
- **SwiftSoup backend** (`packages/swiftsoup/`): SwiftSoup 2.7+ compiled to dynamic library via SPM; `SwiftSoupBridge` @objc class in `swift/Sources/SwiftSoupWrapper.swift` wraps SwiftSoup API with handle-based stores (elements, node lists, text nodes); ObjC header generated by SPM build at `.build/.../SwiftSoupWrapper-Swift.h`, minimal header in `swift/SwiftSoupBridge.h`; ffigen generates Dart bindings from the minimal header; `hook/build.dart` compiles Swift for macOS/iOS and registers as CodeAsset
- **SwiftSoup binding regeneration**: `cd packages/swiftsoup && dart run tool/generate_bindings.dart` ‚Äî builds SPM, auto-extracts `SwiftSoupBridge.h` from SPM-generated `SwiftSoupWrapper-Swift.h` (strips Swift macros), then runs ffigen; header is auto-generated, not manually maintained
- **SwiftSoup dylib preload**: `swiftsoup_init()` (`@_cdecl` in Swift, `@ffi.Native` in Dart) must be called before any ObjC class access ‚Äî ffigen ObjC bindings use `objc.getClass()` (`late final`) which resolves before `@ffi.Native` functions trigger `@DefaultAsset` dylib loading; `SwiftSoupParser` constructor calls `_ensureLoaded()` to force this; `generate_bindings.dart` extracts plain C function declarations from the SPM header alongside the `@interface` block
- **JAR discovery**: `JreManager._findJsoupJar()` checks exe-relative paths ‚Üí walks upward from `Directory.current` for `.dart_tool/hooks_runner/shared/jsoup/build/jsoup-<version>/`; no env var fallbacks; version constant in `lib/src/jsoup_version.dart`
- **MSYS2/MinGW CMake**: Windows backslash paths cause `Invalid character escape` in CMake; use `/c/...` format (not `C:\...`); `jni:setup` expects MSVC `Debug/` layout ‚Äî `generate_jni_bindings.dart --jni-setup` works around this by rescuing `libdartjni.so` from the temp dir
- **JNI test prereq**: Run `melos run jni_setup` (or `dart run tool/generate_jni_bindings.dart --jni-setup` from `packages/jsoup/`) ‚Äî uses bundled JDK and handles MSYS2/MinGW CMake path issues; raw `dart run jni:setup` fails under MSYS2
- **jsoup tests**: `@TestOn('vm')` required (native-only); create fresh `Jsoup()` in `setUp`, call `jsoup.dispose()` in `tearDown`; scraper backend is default on Windows/Linux ‚Äî no JRE/JNI setup needed; JNI backend tests tagged `@Tags(['jni'])` and skipped by default via `dart_test.yaml` ‚Äî run separately: `dart test -t jni --run-skipped` (avoids Windows VEH crash when combined with scraper tests)
- **Windows DLL loading**: `dartjni.dll` depends on `jvm.dll` at load time; `JreManager` pre-loads `jvm.dll` by full path (`DynamicLibrary.open(jvmLibPath)`) before `Jni.spawnIfNotExists` ‚Äî no PATH modification needed
- **JVM is process-global**: Use `Jni.spawnIfNotExists` (not `Jni.spawn`) ‚Äî child isolates share the JVM created by the main isolate; `JreManager.ensureInitialized()` is safe in child isolates (early return via `GetModuleHandleW` when jvm.dll already loaded)
- **Web backend** (TeaVM): `TeaVMParser` in `lib/src/web/teavm_parser.dart` ‚Äî loads TeaVM-compiled Java Jsoup via Blob URL + `importScripts` (Worker-only); the UMD module exports all bridge functions on `self`
- `platform/parser_web.dart` is the web branch of the conditional export ‚Äî MUST NOT import files that use `dart:io` or `dart:ffi` (transitive imports included)
- `host_store.dart`: `HtmlElementResource`, `HtmlElementsResource`, `BytesResource`, `HttpRequestResource`, canvas/image/font resources
- **Response headers**: `HttpRequestResource.responseHeaders` populated by `net::send` ‚Äî native path: JSON-serialized through `WasmSharedState` header slots (`headersPtrSlot`/`headersLenSlot`); web path: parsed from `XMLHttpRequest.getAllResponseHeaders()`; `AsyncHttpDispatch` typedef returns `({int statusCode, Uint8List? body, Map<String, String> headers})`
- **Web JS interop**: `dart:js_interop_unsafe` is required for `JSObject.getProperty`/`setProperty`; `JSNull`/`JSUndefined` are not types ‚Äî use `jsValue.isUndefinedOrNull` instead; extension types with setters need matching getters (`avoid_setters_without_getters`)
- **Aidoku web WASM**: Runs in a Dart isolate (`lib/src/web/wasm_worker_isolate.dart`) compiled to a Web Worker
  - Reuses `buildAidokuHostImports()`, `HostStore`, `WasmRunner` (web), `Jsoup()` (TeaVMParser) ‚Äî unified with native
  - `Isolate.spawn(wasmWorkerMain, initMap)` + `SendPort`/`ReceivePort` with Map-based messages (structured-cloneable)
  - Sync XHR via `dart:js_interop` (`_JSXMLHttpRequest`) ‚Äî required because WASM host imports must return synchronously
  - `LazyWasmRunner` (`lib/src/wasm/lazy_wasm_runner.dart`) breaks circular dependency ‚Äî shared by both native `wasm_isolate.dart` and web `wasm_worker_isolate.dart`
  - Rate limiting enforced in-worker via `RateLimiter` + busy-wait before each sync XHR (unlike native where it's on the main isolate, because web HTTP is synchronous)
- **Web JS interop arity**: dart2js `.toJS` creates fixed-arity JS functions; WASM imports need variable-arity. `_varArgsFactory` in `packages/web_wasm_runner/lib/src/web_wasm_runner.dart` uses `eval` to create wrappers that forward `arguments` as JSArray to a 1-arg Dart bridge. Requires `eval` (already needed since WASM uses `wasm-eval`)
- **TeaVM web bundle** (`packages/jsoup/`): `tool/build_teavm.dart` compiles `tool/teavm/` (Maven + TeaVM 0.13.0 + Jsoup 1.18.3) ‚Üí `lib/src/web/teavm_bundle.dart` (567 KB minified JS as const string); `package:jsoup/teavm.dart` exports `teavmJsoupJs`
- **TeaVM build requirements**: JDK 17+ and Maven 3.x; `pom.xml` needs explicit `teavm-core` dep (contains `PlatformDetector`); uses `@JSExport` (not `@Export`); ADVANCED optimization < FULL size
- **TeaVM Java bridge** (`tool/teavm/src/main/java/.../JsoupBridge.java`): 54 `@JSExport` static methods wrapping Jsoup API with `HashMap<Integer, Object>` handle store; method renames: `id`‚Üí`elementId`, `get`‚Üí`getAt`, `remove`‚Üí`removeElement`, `dispose`‚Üí`disposeAll` (mapped back in `TeaVMParser`)
- **`code_assets` `OS` class**: No `web` constant ‚Äî only native OS values (android, iOS, linux, macOS, windows, fuchsia); `buildCodeAssets` is already false for web targets, so build hooks return early
